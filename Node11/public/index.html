“Trabajamos en una empresa que se encarga de la reparación de computadoras, donde el usuario puede agendar una cita para que pueda dejar su equipo de cómputo, y el personal de la empresa actualizara cuando este listo el equipo del cliente. La empresa nos contrato como desarrolladores de backend para crear una API que sea capaz de poder agendar citas para los usuarios, y que los empleados puedan actualizar cuando los equipos ya estén listos para que los recojan”
Inicializar un proyecto con npm (para poder instalar paqueterías) Instalar express y nodemon (la ultima como dependencia de desarrollo) Crear un servidor de express, corriendo en el puerto de su gusto Conectarse a su base de datos en PostgreSQL
Crear los siguientes modelos basados en el siguiente diagrama:
Para el status del modelo user, asignar como valor default available.
Para el status del modelo repairs, asignar como valor default pending.
Definir los siguientes endpoints /api/v1/users
HTTP VERB URL Descripción GET / Obtener la lista de los usuarios en la base de datos GET /:id Obtener un solo usuario dado un id POST / Crear un nuevo usuario, se debe proporcionar por el req.body (name, email, password, role), el role (rol) puede ser client o employee PATCH /:id Actualizar los datos de un usuario dado un id, solo puede actualizar su name y email DELETE /:id Deshabilitar la cuenta de un usuario Para los endpoints con /:id, se debe validar que el usuario exista, en caso de que no, enviar mensaje de error NOTA: De momento, no se preocupen por la parte de validación de roles o inicio de sesión, esto se cubrirá en el futuro. /api/v1/repairs
HTTP VERB URL Descripción GET / Obtener la lista de equipos pendientes (pending) de reparar GET /:id Obtener un equipo pendiente de reparar por su id POST / Crear una cita, se debe incluir en el req.body lo siguiente (date, userId) El userId siendo el id del usuario quien solicita la reparación. PATCH /:id Actualizar el status de una reparación ha completado (cambiar status a completed) DELETE /:id Cancelar la reparación de un usuario (cambiar status a cancelled) Para los endpoints con /:id, se debe validar que el servicio exista con status pending, en caso de que no, enviar mensaje de error.
